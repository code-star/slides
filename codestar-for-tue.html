    • 



Organisations need a consistent approach to deal with these challenges.

    • 









Reactive systems




Do not confuse with


    

  • Reactive Programming
    • 

  • Functional Reactive Programming (but we use it too!)
    • 

  • React (a JS-library)
    • 





    

  • Reactive programming is the practice of programming with asynchronous event streams
    • 

  • Works with continues time
    • 

  • Reachibg of reactive streams via FP
    • 

  • RS – interaction between the individual parts
    • 

  • Possible to write a single application in a reactive style
    • 









Main characteristics




Check out the official website: www.reactivemanifesto.org






Responsive


    

  • Responses in a timely manner
    • 

  • Provision of rapid and consistent response times
    • 

  • Interaction, simplified error handling
    • 





    

  • Giving users effective interactive feedback.
    • 

  • Encourages interaction, simplifies error handling
    • 









Resilient


    

  • System is resposible after a failure
    • 

  • Achived via replication, containment, isolation and delegation
    • 





    

  • Ideally should be self-recovered
    • 

  • This requires component isolation and containment of failures
    • 

  • To avoid failures spreading to neighbouring components—resulting catastrophic, cascading failure scenarios
    • 









Elastic


    

  • System stays responsive under varying workload
    • 

  • Achived via increasing and decreasing of alocated resources, absence of bottlenecks, distribution of workload
    • 





    

  • Elasticity is about responsiveness under load
    • 

  • Take advantage of the promises of cloud computing: allowing systems to be resource efficient, cost-efficient, environment-friendly etc
    • 









Message Driven


    

  • What makes the reactive systems unique
    • 

  • In-system communication via asynchronous message-passing
    • 



It ensures loose coupling and establish boundaries between components.




    

  • Responsiveness, resilience, elasticity are standard nowdays
    • 

  • Relies on interactions via asynchronous message-passing that estalish boundaries
    • 

  • Communication in non-blocking manner
    • 

  • Boundaries, Location Transparency (topology is not static anymore)
    •

Pronto as 
 a Reactive System




Message Driven


    

  • Messages are delivered via Kafka and RabbitMQ
    • 

  • In case of failure the messages are not lost and services are not blocked
    • 





    

  • 









Resilient


    

  • All the services can work independently
    • 

  • Any service can fail
    • 

  • Messages about failures
    • 

  • TODO: self-healing system
    •

Elastic


    

  • Partially ensured by the messaging systems (Kafka or RabbitMQ)
    • 

  • Collection of metrics helps in resources allocation
    • 

  • Burst performance via Heroku
    •

Challenges
